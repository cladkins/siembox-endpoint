# SIEMBox Endpoint API Contract

This document is the **shared source of truth** for the HTTP API spoken between
the SIEMBox Endpoint agent (this repo) and the SIEMBox server (`cladkins/SIEMBOX`).
The Go wire types live in [`internal/models/models.go`](../internal/models/models.go)
and must stay in sync with the server implementation.

All endpoints are served by the SIEMBox backend on **port 8421** under the
`/api/edr/` prefix. All request/response bodies are JSON.

## Authentication

- **Enrollment** (`POST /api/edr/agents/enroll`) is unauthenticated but requires
  a valid, unused **enrollment token** in the body. Enrollment tokens are
  generated by an admin in the SIEMBox UI (mirroring how shipper API keys work
  today) and may be single-use or time-boxed at the server's discretion.
- **All other endpoints** require the per-agent API key returned at enrollment:
  - `Authorization: Bearer <agent_api_key>`
  - `X-Agent-ID: <agent_id>`
  - The server stores only a hash of the API key.

## Endpoints

### `POST /api/edr/agents/enroll`
Exchange an enrollment token for a permanent agent identity + initial config.

Request:
```json
{
  "enrollment_token": "string",
  "hostname": "string",
  "os": "linux | darwin | windows",
  "os_version": "string",
  "arch": "amd64 | arm64",
  "agent_version": "string",
  "ip": "string"
}
```
Response `200`:
```json
{
  "agent_id": "string",
  "agent_api_key": "string (store hashed)",
  "config": { /* AgentConfig, see below */ }
}
```

### `POST /api/edr/agents/:id/heartbeat`
Liveness ping. Server updates `last_seen` and returns the desired config
version so the agent knows whether to re-pull config.

Request: `{ "status": "online", "agent_version": "string" }`
Response `200`: `{ "config_version": 3 }`

### `GET /api/edr/agents/:id/config`
Return the current `AgentConfig`.

```json
{
  "config_version": 3,
  "heartbeat_interval_seconds": 60,
  "config_poll_interval_seconds": 300,
  "inventory_interval_seconds": 3600,
  "vuln_scan_interval_seconds": 86400,
  "enabled_modules": ["inventory", "vuln", "detect"],
  "rule_set_version": 12,
  "rules": ["<sigma rule yaml>", "..."],
  "yara_rules_version": 7
}
```

`yara_rules_version` increments whenever the server's curated YARA signature
bundle changes. When it exceeds the agent's applied version, the agent fetches
the bundle from `GET /api/edr/agents/:id/yara` and reloads detection. Bump
`config_version` whenever `yara_rules_version` changes so the heartbeat
triggers a config re-pull.

### `GET /api/edr/agents/:id/yara`
Return the server's curated YARA signature bundle as raw rule text
(`Content-Type: text/plain`). This is the concatenation of the chosen
YARA-Forge tiers (Core + Extended) plus any custom rules. The agent appends its
embedded baseline automatically, so the server need only return its own rules;
an empty body is valid. Auth: agent key (Bearer + `X-Agent-ID`).

The agent calls this only when `yara_rules_version` increases, writes the bundle
to `<state-dir>/yara/siembox.yar`, and restarts its osquery YARA scanning to
apply it. Response: `200 OK` with the rule text.

### `POST /api/edr/inventory`
Upsert the endpoint as an **asset** of type `endpoint`. The server maps
`agent_id` → asset so endpoint data correlates with existing log/Nuclei data.

```json
{
  "agent_id": "string",
  "inventory": {
    "hostname": "string",
    "os": "string",
    "os_version": "string",
    "arch": "string",
    "ip": "string",
    "mac": "string",
    "agent_version": "string",
    "software": [{ "name": "openssl", "version": "3.0.2", "source": "dpkg" }],
    "collected_at": "RFC3339"
  }
}
```
Response: `202 Accepted`.

### `POST /api/edr/events`
Batch of detection/telemetry events. **Detections** (`type: "detection"`) are
normalized server-side into the existing **alerts** pipeline; telemetry may be
stored in the parsed-logs store.

```json
{
  "agent_id": "string",
  "events": [
    {
      "id": "uuid",
      "timestamp": "RFC3339",
      "type": "detection | telemetry",
      "severity": "low | medium | high | critical",
      "title": "Suspicious process spawned shell",
      "rule_id": "sigma-xyz",
      "rule_name": "Shell from web server",
      "source": "osquery:process_events",
      "fields": { "pid": 1234, "cmdline": "..." }
    }
  ]
}
```
Response: `202 Accepted`.

### `POST /api/edr/vulnerabilities`
Result of one local vuln scan. Upserted into the existing **vulnerabilities**
table, linked to the endpoint asset.

```json
{
  "agent_id": "string",
  "scan_started_at": "RFC3339",
  "scan_completed_at": "RFC3339",
  "vulnerabilities": [
    {
      "cve": "CVE-2024-1234",
      "package": "openssl",
      "installed_version": "3.0.2",
      "fixed_version": "3.0.13",
      "severity": "high",
      "cvss": 7.5,
      "description": "string",
      "source": "grype"
    }
  ]
}
```
Response: `202 Accepted`.

## Server-side mapping summary

| Agent input        | SIEMBox table        | Notes                                    |
|--------------------|----------------------|------------------------------------------|
| enroll / heartbeat | `edr_agents` (new)   | identity, status, last_seen, config_ver  |
| inventory          | `assets` (reuse)     | `asset_type='endpoint'`, agent↔asset map |
| events (detection) | `alerts` (reuse)     | normalized into existing alert pipeline  |
| vulnerabilities    | `vulnerabilities`    | reuse, linked to endpoint asset          |
| yara (download)    | `edr_yara_bundle`    | curated YARA rule text + version         |

New server tables: `edr_agents`, `edr_enrollment_tokens`, `edr_yara_bundle`.
Everything else reuses the existing SIEMBox data model so the current
alert-triage and vulnerability dashboards work for endpoints with minimal new
UI.

## Offline behavior

The agent spools `events`, `vulns`, and `inventory` payloads to disk when the
server is unreachable and replays them in FIFO order on reconnect. The server
must therefore treat these endpoints as **idempotent-friendly**: events carry a
stable `id`, and inventory/vuln upserts should be safe to receive more than once.
